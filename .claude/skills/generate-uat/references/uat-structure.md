# UAT Plan Structure Reference

Use this as the skeleton for every UAT plan generated by the generate-uat skill.

---

## File naming

`artifacts/{project-slug}-uat-plan.md`

---

## Document structure

```markdown
# UAT Plan — {Project Name}
### {Client} × {Partner / Delivery Team}

**Project:** {name}
**Repo:** `{owner/repo}`
**Client:** {end client}
**Partner:** {agency or partner, if applicable}
**UAT Lead (Delivery):** {delivery lead name}
**UAT Lead (Client):** {name, role — TBD if unknown}
**UAT Scope:** {what phases / features are in scope}
**UAT Target Start:** {date}
**UAT Target Complete:** {date}
**Go-Live Target:** {date}
**Version:** 1.0 (based on codebase review {date})

---

## Scope

State clearly:
- What IS being tested (phase, features)
- What is NOT being tested (name it explicitly — prevents scope creep)

---

## Current Status & Active Blockers

Table: Component | Status (Stable / Partial / ❌ Blocked) | UAT Risk (Low / Medium / High)

Call out any active blockers that must be resolved before UAT can complete.

---

## Pre-UAT Prerequisites

Checklist of everything that must be true before Day 1:
- Environment variables / feature flags that must be set
- API credentials and access that must be confirmed
- OAuth scopes that must be granted
- Test data that must be prepared
- Access that client/partner must have
- Open issues that must be resolved before UAT starts

### Test Data Requirements

Table: what test data is needed, in what format, who prepares it.

---

## Acceptance Criteria

Numbered table: # | Criterion (measurable, specific)

Each criterion maps to one or more test cases below.
Write criteria that can be objectively pass/failed — not "works correctly".

---

## Test Cases

Organised by module / feature area. Each module is a ### heading.

For each test case:

| TC# | Test Case | Steps | Expected Result | Pass/Fail | Notes |
|---|---|---|---|---|---|

Rules:
- Use real endpoint paths, real field names, real status values from the codebase
- Reference issue numbers for known bugs being tested
- Include at least one error handling / resilience test per integration
- Include idempotency tests where webhooks or duplicate processing is a risk
- Include a "happy path" AND at least one "failure path" per major flow
- For AI/ML/extraction: include accuracy measurement test cases with defined thresholds

---

If the system uses AI, ML, OCR, or extraction:

Define measurable pass criteria agreed with client before UAT starts:

| Metric | Acceptance Threshold |
|---|---|
| {metric} | {threshold} |

---

## Defect Handling

Standard severity table:

| Severity | Definition | SLA | Example |
|---|---|---|---|
| P1 — Blocker | Core flow broken; data loss | Fix before sign-off | {example from this system} |
| P2 — Critical | Degraded; workaround possible | Fix before sign-off | {example} |
| P3 — Minor | Cosmetic; edge case | Fix or accept in writing | {example} |

Process:
1. Defect found → log with TC#, steps, severity, screenshot/log
2. Assign to delivery team
3. Fix verified before re-test
4. All P1/P2 resolved before sign-off

---

## UAT Schedule

Day-by-day table: Day | Activity | Owner

Default 4-7 days depending on complexity.

---

## Exit Criteria

Checklist — all must be ticked before sign-off:
- All P1 and P2 defects resolved and re-tested
- All test cases have Pass or Accepted-Risk result
- Any write-back / integration verified at the destination system (not just frontend)
- Client UAT lead confirms in writing
- Any feature flags / env vars confirmed for production

---
```

## Tone & style notes

- Use real names from the codebase (endpoint paths, field names, status strings)
- Be specific: "Reply 'YES' or 'Y' or 'CONFIRM'" not "Send a confirmation reply"
- Flag risks as risks — don't soften them
- Keep test cases executable by someone who hasn't read the code
- Note issue numbers inline: "Tests issue #85"
